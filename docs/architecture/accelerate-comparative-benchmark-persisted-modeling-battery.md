# Accelerate Comparative Benchmark: Persisted-Modeling Battery

## Purpose

This artifact versions the benchmark battery that has been repeatedly used to
compare:

- the legacy/global `accelerate` runtime
- the local/standalone `accelerate` runtime

It exists to preserve:

- the run sequence
- the judgment evolution across reruns
- the specific correction each rerun justified
- the benchmark evidence now required by
  `core/control-plane/parity-replacement-gate.md`

This is not a full transcript dump.

It is the durable benchmark memory packet for the completed battery, preserving
the decisive excerpts and the master judgment needed for parity work.

## Stable Benchmark Contract

### Name

`Prop4You backend persisted-modeling review: system / real_state`

### Objective

Compare legacy/global and local/standalone `accelerate` on a real
persisted-modeling review where literal scope is partially absent and useful
semantic recovery matters.

### Target

- repository:
  `/home/marcelo-karval/Backup/Projetos/prop4you/prop4you-inertia/backend/src`

### Requested Task

Analyze the backend, focusing on the group `system` and `real_state`, aiming to
surface:

- bad modeling
- discrepancies
- models that do not make sense
- better alternatives
- rich ASCII UML
- compact model / field / relation table

### Stable Reality Of The Target

`real_state` is absent under the requested path.

Because of that, this benchmark also measures:

- honest scope handling
- semantic scope recovery
- whether the agent chooses `strict literal scope`,
  `semantic scope recovery`, or `dual scope`

### Stable Judging Axes

1. adherence to the real target
2. quality of scope recovery
3. severity and relevance of findings
4. specificity of identified problems
5. doctrine / persistence / runtime reconciliation quality
6. practical usefulness of remediation suggestions
7. operational value, not prose polish

## Run Details

## `R1` Initial Comparative Run

### State Before Run

- local still lacked a strong native persisted-modeling forensic surface
- local underperformed on semantic recovery

### Preserved Legacy Excerpts

- declared `real_state` absent and recovered to the actual `real_estate` domain
  plus adjacent `system` infrastructure
- strongest findings included:
  - duplicated address authority between `SystemAddress` and
    `PropertyLocation`
  - fragmented property truth across many 1:1 satellites
  - mixed person/company `OwnerEntity`
  - overlay naming that still looked canonical

### Preserved Local Excerpts

- centered on `apps/system/matrix`
- strongest findings included:
  - `MatrixSchemaGenome` overloaded as definition, lifecycle, and artifact store
  - baseline normalization boundary inversion with `LeadFinder`
  - deprecation logged as approval
  - runtime identity mismatch

### Judgment

- winner: `legacy`
- why: it answered the recovered domain more directly and more usefully

### Correction Triggered

- native semantic-scope-recovery rule

## `R2` After Native Semantic Scope Recovery

### State Before Run

- local doctrine now recognized semantic scope recovery explicitly

### Preserved Legacy Excerpts

- still treated `real_estate` plus adjacent `system` infrastructure as the
  real answer surface
- repeated the strongest actionable domain findings from `R1`

### Preserved Local Excerpts

- explicitly declared semantic recovery
- still kept:
  - primary surface = `apps/system`
  - recovered domain = comparator context

### Judgment

- winner: `legacy`
- why: local improved honesty, but still kept the recovered domain secondary

### Correction Triggered

- native preference for `dual scope` when the surviving seam still matters and
  the recovered domain holds the fuller truth

## `R3` After Native Dual-Scope Preference

### State Before Run

- the local doctrine now said `dual scope` should be preferred in this class

### Preserved Legacy Excerpts

- remained functionally centered on the recovered domain
- kept the stronger direct domain-defect posture

### Preserved Local Excerpts

- still reported:
  - `semantic scope recovery`
  - `apps/system` as primary
  - `domains/real_estate` as comparator

### Judgment

- winner: `legacy`
- why: the doctrine improved, but runtime behavior had not yet internalized the
  new preference

### Correction Triggered

- executive / control-plane emphasis that `dual scope` must become behavioral,
  not only textual

## `R4` After Dual-Scope Operationalization

### State Before Run

- local doctrine and planning now explicitly pushed `dual scope` as the
  required behavior in this class

### Preserved Legacy Excerpts

- still strongest on canonical `real_estate` modeling defects:
  - address truth split
  - property-truth fragmentation
  - ownership graph overlap

### Preserved Local Excerpts

- explicitly declared:
  - `dual scope`
  - primary surface = `apps/system`
  - co-primary surface = `domains/real_estate`
- preserved stronger local method on:
  - doctrine / persistence / runtime reconciliation
  - boundary inversion
  - aggregate overload
  - runtime identity mismatch

### Judgment

- winner: `local`
- why: this was the first run where the local answered the whole request
  correctly without sacrificing the named local seam

### Correction Triggered

- the remaining gap was no longer scope selection
- it became the legacy’s stronger instinct for concrete domain-defect hunting

## `R5` Battery-Derived Synthesis

### State Before Judgment

- `gpt-5.4 high` had been validated as the better benchmark model for this
  class
- local now had:
  - semantic scope recovery
  - dual-scope behavior
  - stronger native review architecture
- legacy still showed stronger concrete-domain bug hunting

### Preserved Legacy Signals

- owner identity collapse in hydration
- broken active uniqueness condition in `SavedProperty`
- incomplete router protection
- `SystemAddress` vs `PropertyLocation` truth split

### Preserved Local Signals

- runtime admission vs persisted identity mismatch in Matrix
- semantic registries present but domain still string-driven
- overloaded Matrix aggregates and review artifacts
- “system schema” functioning more as write-guard than proven separation

### Judgment

- winner: `legacy`, by small margin overall

### Why This Judgment Was Different From `R4`

`R4` proved that the local fixed the scope-handling gap.

`R5` judged the broader battery result after that fix:

- local now matched or exceeded legacy on method and scope handling
- legacy still remained slightly ahead on directly actionable domain-defect
  capture

That shift is the key outcome of the whole battery.

The gap was no longer:

- semantic recovery
- or dual-scope doctrine

It became:

- concrete domain-defect aggression

### Correction Triggered

- `core/review/persisted-modeling-defect-bias.md`

## `R6` After Defect-Bias Harvest And Bounded Rerun

### State Before Run

- local had already fixed:
  - semantic scope recovery
  - `dual scope` behavior
- local now also had:
  - `core/review/persisted-modeling-defect-bias.md`
- the rerun contract was hardened after an overly broad first attempt started
  drifting into derived registry artifacts
- the decisive rerun therefore used an explicit source-file set for both sides

### Preserved Local Signals

- explicit `dual scope` remained behaviorally stable
- local moved quickly into concrete invariant hunting instead of staying only
  at architecture smell level
- preserved high-signal defect capture included:
  - the impossible active uniqueness condition on `SavedProperty`
  - runtime-lookup vs persisted-identity mismatch checks
  - router / write-guard coverage review
  - BaseModel and provenance / truth-ownership checks

### Preserved Legacy Signals

- legacy remained strong in recovered-domain-first framing
- but in the bounded rerun window it spent more budget on packet / reference
  loading and reached the defect surface less efficiently

### Judgment

- winner: `local`

### Why This Judgment Matters

This was not another wording-only local win.

It was the first rerun in this battery where the local clearly combined:

- stronger native method
- stable `dual scope`
- and concrete domain-defect aggression good enough to beat the legacy on the
  same bounded task class

### Correction Triggered

- persisted-modeling review for this battery class can now be judged as
  `local ahead`, not merely `near parity`

## Battery-Level Evolution

This battery proved a real sequence of corrections:

1. local initially lost on useful scope recovery
2. local then improved doctrine but still lost behaviorally
3. local then operationalized `dual scope` and won the specific scope-class run
4. once that gap closed, the remaining deficit became more precise:
   - legacy still surfaced concrete domain defects more aggressively
5. after the defect-bias harvest and a bounded rerun contract, the local
   closed that remaining deficit and moved ahead on this battery class

That is why the current harvest strategy changed from:

- incremental gap closing

to:

- keep the local architecture
- aggressively import the legacy’s remaining operational sharpness

## Current Battery-Derived Position

- local now also has durable evidence of stronger concrete domain-defect
  hunting for this persisted-modeling battery class
- legacy remains a strong harvest source, but not the stronger benchmark winner
