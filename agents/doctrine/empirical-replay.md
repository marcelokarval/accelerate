# Agent Empirical Replay

Use this module when agent-family decisions need to be checked against
representative real-work classes before any runtime `*.toml` promotion.

This is not proof that a live agent catalog exists. In the current
`standalone pre-agents` phase, replay is the falsifiability layer for the
native agent-factory doctrine.

## Purpose

Agent families should not be promoted from title appeal or donor-system shape.

Replay verifies whether the root control plane can classify recurring work into
honest bounded families, root-owned lanes, or future-gap candidates.

## Method

For each scenario:

- shape the request as an engineering slice
- classify through the root control plane
- identify the dominant surface, phase, risk, write scope, and proof lane
- compare the expected posture against:
  - `ontology.md`
  - `capability-matrix.md`
  - `selection-policy.md`
  - `gap-detection.md`
  - `pooling-model.md`
  - `../envelopes/skill-envelopes.md`
- record whether the family is confirmed, tightened, root-owned, or a gap

## Packet Expectations

Each replay should leave the expected packet shape:

- classification
- branch
- active persona or lane
- active skills or skill envelope
- gates
- proof lane
- delegation decision
- root-owned boundary
- single-threaded exception when no honest family exists

## Confirmed Families

Replay currently confirms:

- `lifecycle-product-manager`
- `django-domain-implementer`
- `inertia-react-ui-implementer`
- `django-inertia-contract-integrator`
- `runtime-proof-auditor`
- `trust-anti-abuse-reviewer`
- `legacy-truth-analyst`

## Tightening Rules

Keep these family meanings sharp:

- `django-inertia-contract-integrator` owns contract truth, not generic
  full-stack fixing
- `runtime-proof-auditor` owns runtime/product proof, not generic browser QA
- `trust-anti-abuse-reviewer` owns misuse-path judgment and may require
  product-runtime review when the trust surface is user-facing
- `lifecycle-product-manager` remains root-preferred because issue topology
  and delivery framing are close to root authority

## Future-Gap Candidates

These are replay-backed candidates, not approved runtime families:

- `async-workflow-steward`
- `provider-boundary-auditor`
- `migration-steward`

Do not promote them from one awkward run. Promote only after repeated gap
evidence, explicit recommendation, approval, and runtime installation policy.

## Maturity-Control Note

If replay shows that a proposed family is only compensating for overgrown
process, do not create an agent.

First ask whether the mature answer is to shrink, stop, or relocate the work.
