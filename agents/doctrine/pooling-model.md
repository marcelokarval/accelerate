# Agent Pooling Model

Use this module when the root must reason about the bounded-family pool as a
living system rather than a flat list of possible roles.

## Core Rule

The pool is a control-plane asset.

It is not a marketplace catalog and it is not a reason to use agents whenever
one sounds approximately relevant.

## Pool Layers

### 1. Manager lane pool

Root-owned macro lanes, not bounded families:

- executive routing
- technical lane governance
- lifecycle lane governance
- design-contract lane governance
- proof lane governance
- trust lane governance

### 2. Core family pool

The current intended recurring families:

- `lifecycle-product-manager`
- `django-inertia-technical-planner`
- `django-domain-implementer`
- `inertia-react-ui-implementer`
- `django-inertia-contract-integrator`
- `runtime-proof-auditor`
- `trust-anti-abuse-reviewer`

### 3. Risk detection pool

Continuous root lenses:

- decomposition-risk detection
- staffing-risk detection
- contract-risk detection
- proof-risk detection
- trust-risk detection
- rollout-risk detection

### 4. Closure blocker pool

Formal blocker classes, not families:

- proof blocker
- trust blocker
- lifecycle blocker
- contract blocker
- rollout blocker

### 5. Issue topology pool

Structural work shapes:

- single issue
- sibling issues
- parent plus child
- parent plus child plus review lane

### 6. Evidence pool

Proof classes:

- implementation evidence
- QA evidence
- browser truth
- persistent regression proof
- forensic closure evidence

### 7. Conditional / specialist pool

Valid but non-default specialties:

- `legacy-truth-analyst`

### 8. Future-gap pool

Named placeholders for possible future families:

- async workflow stewardship
- provider-boundary auditing
- migration stewardship

## Fit Scoring Dimensions

Before delegating, score candidate families on:

- surface fit
- phase fit
- risk fit
- write-scope fit
- integration cost

If integration cost is high and fit is partial, keep the work local or call it
a gap.
