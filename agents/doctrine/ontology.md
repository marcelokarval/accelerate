# Agent Ontology

This document defines the stable bounded-agent families that the standalone
`accelerate` platform may later promote into a governed runtime catalog.

The ontology should stay small, opinionated, and grounded in real workflow
boundaries rather than organization charts or generic developer titles.

## Core Rule

Families must be derived from recurring bounded work that the root control
plane can delegate honestly.

They must not be derived merely from:

- folder names
- technology keywords
- donor-repo job titles

## Root-Owned Boundary

These functions remain root-owned unless a later explicit governance decision
changes the rule:

- global planning authority
- issue topology decisions
- lane opening and blocker decisions
- final integrated review
- final closure

Bounded families are subordinate to that root law.

## Family Design Rules

### 1. Derive from workflow boundaries

Start from recurring branches in the control plane:

- framing and planning
- bounded implementation
- contract integration
- runtime proof
- trust review
- legacy truth extraction

### 2. Prefer strong profile-aware families

The current default distribution is still stack-opinionated. That means family
names may reflect the dominant profile when that improves honesty.

Examples:

- `django-domain-implementer`
- `inertia-react-ui-implementer`
- `django-inertia-contract-integrator`

### 3. Keep the pool small

The initial pool should cover recurring work without role explosion.

If repeated work still has no honest owner, that is a gap signal, not a reason
to invent many weak families.

## Current Intended Families

### `lifecycle-product-manager`

Owns execution-ready issue framing, acceptance shape, and parent/child hygiene.

It does not own implementation or final closure.

### `django-inertia-technical-planner`

Owns stack-aware decomposition, execution order, and contract-sensitive
technical planning for the current default profile.

It does not absorb implementation or root integration authority.

### `django-domain-implementer`

Owns bounded backend mutation on the Django/domain side.

### `inertia-react-ui-implementer`

Owns bounded frontend mutation on the Inertia/React/UI side.

### `django-inertia-contract-integrator`

Owns backend/frontend contract truth when the boundary itself is the problem.

### `runtime-proof-auditor`

Owns browser/runtime proof for user-facing flows after implementation.

### `trust-anti-abuse-reviewer`

Owns abuse-sensitive and trust-sensitive review for user-driven flows.

### `legacy-truth-analyst`

Owns bounded extraction of legacy or donor-system truth when adaptation is
required.

## Current Phase Note

This ontology is already architecturally real in `standalone pre-agents`.

That does not mean the families are already promoted runtime agents.
