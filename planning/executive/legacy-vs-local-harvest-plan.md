# Executive Plan: Legacy vs Local Side-By-Side Harvest

## Governing Scenario

The standalone `accelerate` repo is still in the `pre-agents` phase and is not
yet consistently superior to the legacy/global runtime in real comparative
tests.

The current diagnosis is no longer “the local lacks concepts.” It is:

- the legacy/global runtime still carries the densest and most coercive
  operational method
- the local/standalone repo already has most of the right architectural homes
- but the local still under-selects, under-enforces, or under-specifies parts
  of that method in real runs

Because of that, the next strategic move is no longer incremental gap-closing
alone. It is a **harvest plan**:

- treat legacy as the methodological baseline
- pull almost everything operationally good into the local repo
- preserve the standalone layered architecture while increasing method density
- only reject or reshape legacy material when it conflicts with the local
  layered architecture or the `pre-agents` reality

The completed benchmark battery made the next step even more explicit:

- keep the local organization
- stop optimizing mainly for cleanliness
- import legacy-grade concrete defect aggression by default

The battery-derived harvest matrix now lives in:

- `planning/executive/legacy-benchmark-battery-harvest-matrix.md`

## Recommended Future Agent Candidates

When the repo reaches the agentization stage, this harvest plan suggests:

- `legacy-harvest-auditor`
  - compares one legacy surface against its local native equivalent and emits a
    transplant recommendation packet
- `benchmark-orchestrator`
  - runs the comparative benchmark contract and scores outcome by method, not
    prose
- `parity-gate-reviewer`
  - decides whether a surface is still `converging`, `near parity`, or ready
    for `local at parity`

## Explicit Non-Goals

- rewriting the local repo to look structurally like the old tree
- treating every local difference from legacy as automatically wrong
- preserving legacy coupling to historical runtime or agent assumptions that do
  not fit the `pre-agents` phase
- continuing an overly timid slice strategy when the baseline we want is
  already obvious

## Harvest Rule

Default assumption:

- if something in legacy repeatedly proves operationally stronger, harvest it
  into local

Exceptions:

- the material depends on future agents as if they already exist
- the material weakens the standalone layered architecture
- the material carries project-specific residue that should live in overlays,
  not core doctrine

This means the burden of proof has inverted:

- before, each local mutation had to earn itself from scratch
- now, legacy-good method is presumed worth harvesting unless a concrete local
  architectural reason blocks it

## Residual Risks Or Open Decisions

- The local may still lag even after harvest if the doctrine is not coercive
  enough at runtime.
- A large harvest can create local bloat if we fail to preserve layer
  ownership.
- Some legacy strength is behavioral and may require stronger benchmark
  invocation contracts, not just richer docs.
- Benchmark memory is now versioned natively, but parity judgments still need
  repeated use of that evidence packet before replacement claims become cheap.
