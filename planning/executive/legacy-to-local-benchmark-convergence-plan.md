# Executive Plan: Legacy-To-Local Benchmark Convergence

## Governing Scenario

The standalone `accelerate` repo is in the `pre-agents` phase and is being
used to converge the local doctrine until it is **superior or equal in
operational value** to the legacy/global runtime, ignoring the future-agent
layer.

The current execution method is benchmark-driven:

- run comparative real-world tests against the same target
- capture divergences between legacy/global and local/standalone behavior
- compare native local docs against legacy/reference doctrine
- rehome only the parts that repeatedly prove operational value
- rerun the benchmark to verify that the change improved actual execution, not
  just local wording

## Selected Workflow Adapter Or Current No-Adapter Reality

This work is currently operating in the standalone no-adapter reality.

- no native workflow backend is yet the permanent execution authority here
- planning and architecture docs remain the governing artifacts
- the control plane is root-governed
- comparative benchmarks are the main proof mechanism for parity claims

## Selected Stack Profile

The active stack profile is the standalone `accelerate` product itself:

- root control plane
- native `core/` doctrine
- inherited `references/` as comparison-only supporting doctrine
- planning-led execution for non-trivial doctrine mutation

## Selected Runtime Adapters

The active runtime shape for this plan is:

- local repo documentary mutation
- paired doctrinal diff against `references/`
- multi-agent comparative benchmark replay
- architecture/gap-matrix based acceptance updates

Browser-proof and persistent E2E are not primary proof lanes for this plan.

## Selected Docs Posture

The documentation posture for this plan is:

- `core/` and repo-root docs are the target authority
- `references/` remain readable as legacy comparison surfaces
- `docs/architecture/accelerate-legacy-to-local-gap-matrix.md` is the live
  residual-gap scoreboard
- `docs/architecture/accelerate-comparative-benchmark-template.md` is the
  benchmark contract
- concrete improvements should be validated by rerunning the same benchmark

## Recommended Adjacent Skills

- `architecture`
- `governance-audit`
- `subagent-governance` when parallel benchmark slices need stricter shaping
- `parallel-agents` when deciding whether concurrent doctrinal inspection is
  safe and useful

## Recommended Future Agent Candidates

When the repo reaches the agentization stage, this plan suggests the following
specialist candidates:

- `benchmark-runner`
  - executes legacy vs local comparisons on the same target
- `doctrine-diff-auditor`
  - compares `references/` against native `core/` surfaces and extracts only
    the operationally winning deltas
- `parity-gate-reviewer`
  - judges whether a surface is still `converging`, `near parity`, or truly at
    parity based on rerun evidence, not directory shape

## Explicit Non-Goals

- claiming parity because the local repo is cleaner
- mirroring legacy doctrine blindly into native local docs
- optimizing for future-agent ergonomics before pre-agents parity exists
- treating benchmark wins as permanent without rerun confirmation
- widening benchmark scope by invention when the target is absent literally

## Next Bounded Slices

1. Promote the first `near parity` workflow-governance surfaces using repeated
   behavioral evidence rather than more document harvest.
   - first queue:
     - `operational calibration`
     - `issue stack`
     - `branch enforcement matrix`
   - authority:
     - `planning/executive/near-parity-promotion-queue.md`

2. Re-run or observe real bounded slices specifically to test those surfaces.
   - success means promotion by repeated operational proof, not by wording diff

3. Use the parity gate after each successful promotion attempt.
   - surface-by-surface promotion only
   - no broad replacement claims from a single win

4. Keep the benchmark battery and gap matrix synchronized after each promotion.
   - benchmark memory remains durable architecture evidence, not session memory

## Residual Risks Or Open Decisions

- The local doctrine may be correct on paper but still under-selected at
  runtime, as seen in the current `dual scope` gap.
- The gap matrix can drift into optimism if it is not recalibrated after real
  reruns.
- Some remaining differences are no longer methodological but behavioral; that
  may require tightening benchmark invocation contracts, not only rewriting
  doctrine.
- The repo still needs explicit versioned benchmark run artifacts if we want
  auditability beyond conversational memory.
