# Parity Decision: Branch Enforcement Matrix

## Purpose

This artifact is the explicit `parity-replacement-gate` packet for the
`branch enforcement matrix` surface.

It records the first focused branch-enforcement benchmark run after the
near-parity promotion queue was created.

  - `core/control-plane/branch-enforcement-matrix.md`

## Why This Surface Matters

This surface governs whether the runtime can:

- refuse fake trivial classification
- load the correct mandatory skills for the active branch
- enforce validation harshness correctly
- block closure when issue/bootstrap/planning hygiene is missing
- expose adjacent gates like `Sidebar / Public Docs Update Gate` when the slice
  crosses discoverability or navigation surfaces

## Benchmark Or Run Evidence

### Benchmark Shape

- focused branch-enforcement benchmark
- target request: backend model mutation + user-facing React mutation +
  docs/navigation discoverability mutation
- explicit absence of governing issue
- explicit under-run validation:
  - only `manage.py check`
  - no `makemigrations --check --dry-run`
  - no `migrate --check`
  - no frontend `type-check`
- judged on:
  - branch classification harshness
  - mandatory skill loading
  - validation blocker accuracy
  - issue/planning blocker accuracy
  - discoverability/docs gate detection

### Local Output Signal

The local side chose:

- non-trivial `issue-driven delivery`
- dual enforcement from:
  - `query / contract-sensitive backend`
  - `visual / artifact-driven frontend`
- explicit living-docs/discoverability scope

Strong points:

- refused `trivial bounded` immediately
- enforced both backend and frontend validation stacks correctly
- surfaced `Sidebar / Public Docs Update Gate` explicitly
- surfaced branch-specific gates with higher fidelity:
  - `Query Shape Proof`
  - `Visual Contract Extraction`
  - `TS/TSX Boundary Audit`
- treated “implementation began before issue bootstrap” as a real workflow
  failure, not only a missing checkbox
- produced a sharper closure-blocker packet for the exact cross-surface shape

### Legacy Output Signal

The legacy side chose:

- non-trivial `issue-driven delivery`
- backend + frontend overlay enforcement
- mandatory issue/bootstrap/planning discipline
- insufficient validation on both backend and frontend

Strong points:

- correctly blocked trivial closure
- correctly enforced issue/bootstrap/planning gates
- correctly treated validation under-run as closure failure

Residual weakness versus local:

- it was slightly less specific on the native cross-surface gate set
- it did not express the exact local branch overlay and closure blockers with
  the same precision as the local side

### Comparative Judgment

Winner for this focused branch-enforcement benchmark:

- `local`, by small margin

Reason:

- both sides blocked closure correctly
- the local side applied the branch matrix more precisely to the concrete
  cross-surface case
- the local answer better reflected the stronger native enforcement clauses
  that were added during the harvest

## Residual Gap Statement

Residual gap:

- no material operational deficit was exposed in this benchmark slice

The remaining caution is not about this surface being under-specified.

It is only the normal caution that one focused benchmark is still narrower than
portfolio-wide repeated evidence.

## Replacement Risk

Low for this surface.

If future runs regress, the likely failure mode would be:

- under-routing a cross-surface slice
- or collapsing the branch to generic `issue-driven delivery` without the
  stronger overlay/gate fidelity shown here

That is a runtime-regression risk, not a current authority gap.

## Decision Proposal

- verdict: `local ahead`

Effect:

- the local branch-enforcement matrix becomes the clear primary authority for
  this surface
- the legacy remains supporting doctrine and historical comparison material,
  not the stronger operational distribution
