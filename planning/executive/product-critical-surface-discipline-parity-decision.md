# Parity Decision: Product-Critical Surface Discipline

## Purpose

This artifact is the explicit `parity-replacement-gate` packet for the
`product-critical surface discipline` surface.

It now records two focused 2x2 reruns:

1. `auth recovery / account access restoration`
2. onboarding-critical visual/compositional pressure

Each rerun used:

- `local + medium`
- `local + high`
- `legacy + medium`
- `legacy + high`

## Surface Under Judgment

- local native surface:
  - `core/review/product-critical-surfaces.md`
  - `core/control-plane/branch-enforcement-matrix.md`
- legacy comparison surface:
  - legacy/global product-critical branch behavior under the same benchmark
    contract

## Why This Surface Matters

This surface governs whether the runtime can:

- escalate auth recovery out of generic runtime work
- require a truly isolated recovery surface
- reject generic private-hub reuse for special-state users
- force backend product contract artifacts before closure
- keep browser truth and anti-abuse lanes open on trust-sensitive recovery work
- reject technically correct but compositionally weak product-critical surfaces
- force wireframe / visual-contract sufficiency before closure

## Native Authority Proof

The local primary written authority now lives in:

- `core/review/product-critical-surfaces.md`
- `core/control-plane/branch-enforcement-matrix.md`
- `core/risk/enforcement-surfaces.md`

The local branch already makes these rules explicit:

- auth recovery is a product-critical surface
- recovery states should not reuse generic private hubs
- special-state users should not roam unrelated private pages
- `anti-abuse-review` is mandatory when the flow is abuse-sensitive
- browser truth remains blocking for closure

## Benchmark Or Run Evidence

### Benchmark Shape A: Auth Recovery

Target run:

- auth recovery / account access restoration flow
- generic private hub reused instead of isolated recovery surface
- temporary allowlist lets special-state users roam unrelated private pages
- backend tests and frontend `type-check` completed
- no `anti-abuse-review`
- no backend `Truth Model` / `Action Model` / `Security Model` / `Page Contract`
- no interactive Chrome DevTools browser truth
- closure attempted as “done because it works locally”

### Local Output Signal: Auth Recovery

#### Local + Medium

- classified the run as `product-critical user surface`
- blocked closure
- required:
  - `anti-abuse-review`
  - `Backend Product Contract Protocol`
  - `Recovery Surface Isolation Rules`
  - browser truth
- explicitly judged recovery-surface isolation as failed

#### Local + High

- preserved the same core judgment
- kept the branch firmly on `product-critical user surface`
- made the issue-driven and backend-contract obligations more explicit

Strong points on the local side:

- clean native treatment of auth recovery as product-critical rather than merely
  “runtime-sensitive”
- explicit rejection of generic private-hub reuse
- explicit rejection of special-state roaming via allowlist
- clear linkage between product-critical closure and missing backend contract
  artifacts

### Legacy Output Signal: Auth Recovery

#### Legacy + Medium

- also classified the run as `product-critical user surface`
- also blocked closure
- also required `anti-abuse-review`, recovery isolation, backend contract
  artifacts, and browser truth

#### Legacy + High

- converged to the same judgment
- used nearly the same branch and gate shape
- no material legacy-only strength appeared in this benchmark

### Benchmark Shape B: Onboarding-Critical Visual Pressure

## Comparative Judgment

Winner after the combined reruns:

- `local`

Operational reading:

- on auth recovery, the local side matched the legacy/global doctrine cleanly
- on onboarding-critical visual pressure, the local side produced the sharper
  and more operationally useful judgment
- taken together, the surface now tilts in favor of the local distribution

## Residual Gap Statement

Residual gap:

- no material doctrinal deficit remains on this surface

The only remaining caution is portfolio-wide:

- product-critical still spans multiple user-surface families
- future reruns should keep checking state-heavy or abuse-sensitive cases

## Replacement Risk

Low for this surface.

If it regresses, the likely failure modes would be:

- downshifting a recovery surface into generic runtime work
- tolerating generic private-hub reuse under delivery pressure
- accepting technically correct but compositionally weak closure on critical
  surfaces

These are runtime-regression risks, not current authority gaps.

## Decision Proposal

- verdict: `local ahead`

Effect:

- the local product-critical surface discipline is now the stronger operational
  authority for this surface family in the standalone repo
- the legacy remains supporting doctrine and comparison material, not the
  stronger operational distribution on this slice

## Next Correction

No doctrinal correction is required before replacement for this surface.

The next useful proof would be:

- a product-critical class with stronger anti-abuse and state-transition
  pressure layered on top
- or a live implementation review instead of a synthetic benchmark-only pass
