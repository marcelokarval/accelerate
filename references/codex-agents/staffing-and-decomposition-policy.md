# Staffing and Decomposition Policy

Use this module when `accelerate` must think like a staffed technical
organization before the runtime agent catalog exists.

## Core Rule

Staffing is a root-level choice about the shape of the work.

It is not just the act of assigning a name to a slice.

## Staffing Ladder

Evaluate in this order:

1. keep at root
2. one family
3. one family plus one review family
4. two complementary bounded families
5. future gap instead of forced fit

## Root-Only Cases

Keep the work at the root when:

- global integration judgment dominates
- lifecycle shape is still unstable
- closure authority is the real work
- the expected gain from delegation is lower than the integration cost

## One-Family Cases

Use one family when:

- one dominant surface exists
- the slice is bounded
- the write scope is clear
- proof can remain local or can be added later as a bounded review lane

## Two-Family Cases

Use two families when:

- one mutates and one proves
- one frames and one executes
- one technical family needs a contract or trust partner

## Budget Rule

Thread budget is a constraint, not an obligation.

Having available capacity does not mean all capacity should be used.

The root should prefer the smallest honest staffing shape that reduces
integration burden.

## Decomposition Rule

Decompose only when:

- the parent outcome is clear
- slices have bounded ownership
- return contracts are clear
- topology is already valid

If these are not true, do not split just to create the appearance of parallel
work.
