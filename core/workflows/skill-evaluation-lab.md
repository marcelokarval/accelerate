# Skill Evaluation Lab

## Purpose

Use this workflow when a skill, workflow lane, adapter rule, trigger
description, or agent recipe needs to improve from evidence rather than taste.

This is the local `accelerate` translation of the strongest GLM / Z.ai
skill-creation pattern: baseline runs, candidate runs, blind comparison,
assertions, analyst pass, and iterative improvement.

## Rule

Workflow evolution must be evaluated before it is promoted.

Do not promote a skill or control-plane rule because:

- it sounds sharper
- it worked in one chat
- it copies a mature external platform
- it improves documentation without behavioral proof

Promote only when the evaluation record shows that the candidate behavior is at
least as useful as the baseline for the target class of work.

## When To Use

Use this lab for:

- new skill creation
- existing skill rewrite
- trigger description tuning
- external skill adaptation
- agent prompt / recipe derivation
- workflow lane promotion
- parity rerun where method behavior is under judgment
- repeated failure pattern where the fix should become doctrine

## Evaluation Packet

Minimum packet:

- `Target`
  - skill, workflow, adapter, agent recipe, or trigger description under test
- `Hypothesis`
  - what should improve
- `Baseline`
  - current local behavior or legacy/reference behavior
- `Candidate`
  - proposed revised behavior
- `Eval Prompts`
  - representative tasks, including edge cases
- `Assertions`
  - objective pass/fail checks where possible
- `Rubric`
  - qualitative criteria when assertions are insufficient
- `Blind Comparison`
  - compare outputs without knowing which side is baseline/candidate when
    feasible
- `Analyst Pass`
  - unblind and explain why the result happened
- `Cost / Time Notes`
  - runtime, token, or coordination cost if relevant
- `Decision`
  - promote, revise, rerun, reject, or keep as pattern-only

## Baseline Vs Candidate

The lab should compare:

- current local behavior vs candidate local behavior
- legacy/reference behavior vs candidate local behavior
- no-skill baseline vs skill-enabled behavior
- current trigger description vs revised trigger description

Pick the comparison that matches the decision being made.

Do not mix comparison targets without naming them.

## Trigger Description Tests

When tuning skill or agent activation descriptions, include both:

- `should trigger`
- `should not trigger`

Use held-out examples when possible.

The goal is not only high activation. The goal is calibrated activation.

Failure modes:

- over-triggering
- under-triggering
- prompt wording overfit
- false confidence from a tiny sample
- description that wins train examples and fails unseen examples

## Blind Comparator Role

When feasible, compare outputs as `A` and `B` before revealing origin.

The comparator should judge:

- task completion
- correctness
- specificity
- risk handling
- proof quality
- user usefulness
- workflow discipline

The comparator should not know which output is local, legacy, baseline,
candidate, or generated by a specific model unless that knowledge is necessary
for the evaluation.

## Analyst Role

After comparison, unblind and analyze:

- what the winning side did differently
- what the losing side missed
- whether the win came from real method quality or accidental verbosity
- whether the candidate introduced ceremony, risk, or brittleness
- what exact local doctrine should change, if any

The analyst must separate:

- behavior improvement
- wording improvement
- artifact organization
- model luck
- overfitting

## Promotion Rule

Promotion requires:

1. durable evaluation artifact
2. explicit decision
3. insertion target
4. no unresolved conflict with existing nearby doctrine
5. workflow-change approval when the promoted rule changes control-plane truth
6. follow-up rerun after landing

## Anti-Bloat Rule

Do not create a new skill, lane, or agent because the lab found an interesting
pattern.

Prefer:

- strengthen existing skill
- add a focused reference
- update runtime adapter contract
- add stack profile guidance
- record pattern-only decision

Create a new workflow lane only when the behavior cannot be cleanly owned by an
existing surface.

## Future Agent Fit

When `*.toml` agents exist, this workflow can be implemented through bounded
roles:

- eval-runner
- blind-comparator
- analyst
- trigger-tuner
- forensic reviewer

The root still owns:

- evaluation target
- acceptance threshold
- promotion decision
- final integration
